# Өгөгдлийн сангийн удирдлагын бичиг баримт болон нууц үгийн бодлогыг хэрэгжүүлэх

## Асуулт
bcrypt болон хуучин hash алгоритмуудын ялгааг тайлбарлаж, яагаад орчин үеийн системүүд bcrypt ашиглах болсон шалтгааныг бичнэ үү.

## Хариулт 

1. Хуучин hash алгоритмууд (MD5, SHA-1):

	•	Маш хурдан ажилладаг.

	•	Нэг ижил нууц үг үргэлж нэг ижил hash утга үүсгэдэг.

	•	Орчин үед rainbow table болон brute force халдлагад амархан өртдөг.

	•	Давс (salt) ихэнхдээ ашигладаггүй эсвэл хамгаалалт багатай.



2. bcrypt алгоритм:

	•	Нууц үгийг hash хийхдээ salt автоматаар ашигладаг.

	•	Удаан тооцоологдохоор бүтээгдсэн тул brute force халдлага хийхэд маш хэцүү.

	•	Cost factor буюу давталтын тоог нэмэгдүүлж аюулгүй байдлыг өсгөж болдог.

	•	Орчин үеийн системүүдэд нууц үг хамгаалахад илүү найдвартай.


Яагаад орчин үеийн системүүд bcrypt ашигладаг вэ?

	•	Хуучин алгоритмууд хэт хурдан тул хакерууд олон сая нууц үгийг хурдан шалгаж чаддаг.

	•	bcrypt нь удаан ажилладаг, salt ашигладаг тул нууц үгийг эвдэхэд илүү хэцүү болгодог.

	•	Тиймээс аюулгүй байдлыг сайжруулахын тулд орчин үеийн системүүд bcrypt-ийг ашигладаг.


## Жишээ

Жишээ нь password123 гэдэг нууц үгийг MD5 алгоритмаар hash хийвэл үргэлж нэг ижил hash гарна. Хэрвээ хоёр хэрэглэгч ижил нууц үг хэрэглэвэл hash нь мөн адилхан байна. Үүнийг хакерууд rainbow table ашиглан амархан тайлж чадна.



Харин bcrypt алгоритм нь salt ашигладаг. Жишээ нь:

	•	password123 + Xy7! → k9Lm2#

	•	password123 + Pq4$ → 8sT1v@



Энд нууц үг ижил боловч salt өөр учраас hash нь өөр гарч байна. Тиймээс хакер нууц үгийг тайлахад маш хэцүү болдог. Иймээс орчин үеийн системүүд bcrypt ашигладаг.




## Дүнгэлт

Товчоор хэлбэл:
MD5/SHA-1 нь "түгжээгүй хаалга" мэт болсон бол, bcrypt нь "цаг хугацааны хоцрогдолтой, түлхүүр нь байнга солигддог сейф" юм. Аюулгүй байдлын алтан дүрэм бол: Нууц үгийг хэзээ ч битгий хурдан hash алгоритмаар хадгал.




### Нэмэлт 

rm .git/index.lock яагаад ийм file үүсгэв

Gemini said
Энэ файл яагаад үүсдэгийг энгийнээр тайлбарлавал, энэ нь Git-ээс таны өгөгдлийг хамгаалж буй "Түр зуурын цоож" юм.

Git нэг дор зөвхөн нэг л чухал үйлдэл (жишээ нь: add, commit, checkout, merge) хийх ёстой байдаг. Хэрэв хоёр процесс зэрэг ажиллавал таны төслийн түүх (history) эсвэл индекс файл гэмтэх аюултай.

index.lock файл үүсдэг гол шалтгаанууд:
Процесс зэрэгцэх (Concurrency):
Таныг терминал дээр git add . гэж бичих үед өөр нэг програм (жишээ нь: VS Code-ийн цаана ажилладаг Git өргөтгөл эсвэл SourceTree) төслийг тань автоматаар индексжүүлж байвал Git энэ файлыг үүсгээд бусад бүх үйлдлийг "цоожилдог".

Гэнэтийн зогсолт (Crash/Interruption):
Өмнөх Git команд ажиллаж дуусаагүй байхад та терминалаа хүчээр хаасан, эсвэл компьютер гацсан тохиолдолд Git "цоожоо" тайлж амжилгүй үлдчихдэг. Ингэснээр дараагийн удаа команд өгөхөд "Өөр процесс ажиллаж байна" гэж андуурдаг.

Том хэмжээний өөрчлөлт:
Маш олон файл зэрэг нэмж байгаа үед Git индекс файлыг шинэчлэхэд хугацаа зарцуулдаг бөгөөд энэ хооронд index.lock файл идэвхтэй байдаг.

Хэзээ устгах вэ?
Устгаж болохгүй үед: Хэрэв таны өөр нэг терминал эсвэл програм дээр Git үнэхээр ажиллаж байгаа бол (жишээ нь: том хэмжээний push хийж байх үед) устгаж болохгүй. Энэ нь өгөгдөл эвдрэхэд хүргэж мэднэ.

Устгаж болох үед: Ямар ч Git команд ажиллаагүй байхад л энэ алдаа гарсаар байвал таны түрүүн хийсэн шиг rm .git/index.lock командыг ашиглан "цоожийг" гараар эвдэж болно.

Таны хувьд git add . хийх үед энэ алдаа гарсан нь:
Магадгүй таны ашиглаж буй код засварлагч (Editor) цаана нь файлыг тань хянаж байсан эсвэл өмнөх команд тань дутуу ажиллаад гацчихсан байсан байх магадлалтай.
